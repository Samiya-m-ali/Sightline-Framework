Sightline

A Framework for Intent-Driven Navigation Design — by Samiya Ali

A structured method for translating user intent — Jobs, Tasks, and Activities — into navigation architecture. Built for product designers, developers, and product managers who need to design, build, and defend navigation decisions.

Context

Why Does Navigation Keep Breaking?

Navigation is one of the hardest parts of product design to get right — and one of the easiest to get wrong incrementally. Teams invest real thought into their information architecture. The challenge is that navigation decisions compound: each feature launch, each new user role, each platform addition adds pressure to a structure that was built for a simpler version of the product.

Over time, patterns emerge. Users struggle to find features that exist because labels reflect internal team structures rather than user goals. Navigation changes with each release cycle because it's anchored to the feature roadmap rather than stable user outcomes. Onboarding takes longer than it should because new users have to reverse-engineer the product's internal logic. Support tickets accumulate around "how do I..." questions that the navigation should have answered on its own.

The underlying challenge isn't a lack of effort — it's a lack of shared structure. The work of connecting user intent to navigation decisions tends to happen informally: in someone's head, on a whiteboard that gets erased, in a design file that diverges from implementation. The thinking happens, but it doesn't persist across the team or survive the pressures of a shipping schedule.

Sightline gives that thinking a structured, repeatable home. It provides a shared language for connecting what users are trying to accomplish to how they navigate your product — so the architecture stays grounded in intent as the product evolves.

Step 01

Define

What are my users' Jobs to Be Done?

Users don't think in pages, menus, or modules. They think in outcomes they're trying to achieve. A navigation system should mirror that intent hierarchy — not your org chart, not your database schema, and not whatever your backend team decided to name the API endpoints.

This model builds on Jobs to Be Done theory (the Job level) and Hierarchical Task Analysis (the Task and Activity levels) — see the Appendix for origins and further reading. Most users have 3–7 Jobs — that's normal and expected. Your navigation system needs to accommodate all of them.

The Three Levels

Job
Changes: Rarely
The overarching outcome a user is trying to achieve. Emotionally resonant, context-independent, and product-independent. This is the why.
Changes when...
The user's role, goals, or life circumstances change. A promotion, a career shift, a new business model — these are the forces that redefine a Job.
Example
"Keep my team's financial health strong so we can grow sustainably" — true whether the user has your software, a spreadsheet, or a paper ledger.
Litmus Test
Ask: "Would this still be true if the product didn't exist?" Jobs are product-independent. If the statement only makes sense inside your product, it's probably a Task.
Task
Changes: Periodically
A discrete, goal-oriented unit of work that advances the Job. Has a clear start and end, requires deliberate effort, and can be planned or delegated.
Changes when...
Workflows, processes, or business needs evolve. New tools get adopted, teams restructure, compliance requirements shift.
Example
"Create and send monthly budget reports" might become "Set up automated reporting dashboards" — same Job, different Task.
Litmus Test
Ask: "Would someone put this on a to-do list?" Tasks are plannable. If you can imagine delegating it to a colleague with a brief explanation, it's a Task. If you'd have to point at the screen, it's an Activity.
Activity
Changes: Frequently
A concrete, atomic interaction within a Task. Happens in real-time, is tool-dependent, and represents the unit of UI interaction design.
Changes when...
The tool or interface is redesigned. A button becomes a drag target, a form becomes a wizard, a manual step gets automated.
Example
"Select date range, pull data, choose template, export PDF" — each would change if the reporting tool was redesigned.
Litmus Test
Ask: "Does this require a specific UI element to exist?" Activities are tool-dependent. If the action would change significantly with a redesigned interface, it's an Activity. Still not sure? Ask: "Would the user plan a session around doing this?" If they'd set aside time, it's a Task. The individual interactions within that session are Activities.
Key Insight

Jobs rarely change — they're tied to the user's role and goals. Tasks change periodically as workflows evolve. Activities change frequently, every time the interface is updated. Your navigation at each level should match this rate of change. If your primary navigation shifts every time you redesign a feature, you've mapped Activities at the Job level.

A Note on Cross-Job Tasks

Some Tasks serve multiple Jobs. "Generate a financial report" might advance both "Keep stakeholders informed" and "Ensure regulatory compliance." Don't force these into a single Job — acknowledge them and move on. We'll address cross-Job Tasks in the Map step.

Defining for Multiple User Types

Most products serve more than one persona. A project management tool serves Project Managers, Developers, and Designers — each with overlapping but distinct Job maps. Don't try to define all personas simultaneously. Instead:

Define one persona at a time. Give each persona its own complete Job → Task → Activity hierarchy. This keeps each map clean and prevents premature averaging.

Then overlay the maps. Once you have 2–3 individual maps, lay them side by side. Look for: shared Jobs, shared Tasks under different Jobs, and persona-exclusive Jobs. Mark these overlaps and divergences — they feed the role-based decisions in Step 04.

Calculate your overlap ratio. If two personas share more than 70% of their Jobs, you're likely building one navigation with different defaults. If they share less than 30%, you may be building two different products. You'll use this overlap ratio to choose your role-based navigation approach in Step 04 — the ratio maps directly to three approaches ranging from shared navigation with different defaults to fully separate applications.

Step 02

Map

How does intent translate to navigation tiers?

You've defined your users' Jobs, Tasks, and Activities. Now you need to map them to navigation tiers. But first, answer a foundational question: what organizes your user's mental model?

The answer determines how your JTBD hierarchy translates to navigation structure. Get this wrong and your navigation will fight the user's thinking — even if every individual component is well-designed.

Choosing Your Navigation Anchor

Users organize their work in one of four dominant mental models. Most products are hybrids, but one model dominates — identify it and let it drive your primary navigation.

Process-first
Jobs = Primary nav
The user thinks in distinct phases or workspaces, each with its own tools and mindset. They say: "I'm in research mode" or "I'm doing my weekly review."
Diagnostic
Do users describe their work as distinct modes or phases? Does each phase involve different tools and a different mindset?
Examples
TurboTax — linear flow through phases. Shopify Admin — Products, Orders, Marketing as distinct workspaces. HubSpot — Marketing, Sales, Service as separate hubs.
MAPPING JOB / Phase ——→ Primary Navigation Always visible "Research" / "Delivery" Top bar, sidebar root | v TASK ——→ Secondary Navigation Within phase context "Run customer interviews" Tabs, sub-sections | v ACTIVITY ——→ In-Page Navigation Contextual / transient "Schedule interview call" Buttons, forms, inline actions
Entity-first
Entities = Primary nav
The user tracks specific things — features, accounts, projects, cases — that persist over time. They say: "How's the Johnson account doing?" The entity is the anchor; Jobs become lenses on that entity.
Diagnostic
Do users refer to their work by name? Do they track entities through multiple Jobs over time? Is their primary question "what's the status of X?"
Examples
Salesforce — Accounts, Contacts, Opportunities as primary objects. Jira — Issues move through workflows. GitHub — Repositories with Code, Issues, PRs as tabs.
MAPPING ENTITY ——→ Primary Navigation Always visible "Feature X" / "Acme Acct" Entity list, entity detail | v JOB / Phase ——→ Secondary Navigation Tabs within entity "Discovery" / "Delivery" Phase tabs, concern tabs | v TASK → ACTIVITY ——→ In-Page Navigation Contextual / transient "Write requirements" Forms, editors, inline actions
Queue/Inbox-first
Incoming stream = Primary nav
The user's work arrives as a stream of items that need processing. The user doesn't go find work; work comes to them. Their first question is "what should I work on next?"
Diagnostic
Does work arrive continuously rather than being initiated by the user? Is the primary interaction processing items from a list? Are items transient (resolved and archived) rather than persistent?
Examples
Zendesk — ticket queues filtered by status and priority. Gmail — inbox with labels and smart categories. Slack — message streams by channel. PagerDuty — incident queue.
MAPPING QUEUE VIEWS ——→ Primary Navigation Always visible "My Queue" / "Unassigned" Filtered lists, inbox views | v ITEM DETAIL ——→ Secondary Navigation Tabs within item "Conversation" / "History" Concern tabs, detail panels | v ACTIVITY ——→ In-Page Navigation Contextual / transient "Reply" / "Escalate" Buttons, quick actions, shortcuts
Activity/Mode-first
Dominant activity = Separate mode
The user's core work requires deep, immersive focus. The product has two worlds: an immersive creation mode and a management/browse mode, connected by an explicit mode-switch.
Diagnostic
Is there one dominant activity that requires full-screen focus? Does that activity account for the majority of usage time?
Examples
Figma — File browser ↔ Canvas. VS Code — Editor dominates. Substack/Ghost — Full-screen editor for writing, dashboard for everything else.
MAPPING MANAGEMENT MODE ——→ Standard Navigation Top bar, sidebar "Browse projects, check Jobs and entities organized metrics, manage settings" using process or entity model ↑↓ explicit mode-switch IMMERSIVE MODE ——→ Minimal / Hidden Navigation Reduced or none "Write the article, Full-screen focus, contextual design the interface" toolbars only, nav recedes
Reactive vs. Proactive Work

Queue/Inbox-first is the model for reactive workflows — the system surfaces what needs attention. The other three assume proactive workflows — the user initiates navigation toward their goals. Reactive navigation needs to bring work to the user; proactive navigation needs to let the user find work. Many products blend both.

Most products are hybrids

Pure instances of any single model are rare. Salesforce is entity-dominant with process elements. A support tool might be queue-first for agents but entity-first for managers. The goal isn't to pick one model exclusively — it's to identify which model dominates and let it drive your primary navigation, then layer secondary models into lower tiers.

Identifying Your Dominant Model

The most common mistake teams make here is projecting the product's current structure onto the user's mental model. Your database might be entity-centric, your roadmap might be process-oriented, and your support queue might be the highest-traffic feature — but none of those tell you how users organize their work. To find the dominant model, you need to look past the product and into the user's world.

Run these four diagnostic checks. If three or more point to the same model, that's your dominant anchor.

1. Listen to the Language
Scan support tickets, user interviews, or session recordings and note how users frame what they're doing. The nouns and verbs people reach for naturally reveal the anchor.
Process-first
"I'm in my weekly review" / "I need to do my month-end close"
Entity-first
"I need to check on the Anderson account" / "What's happening with Project Alpha?"
Queue-first
"Let me see what came in overnight" / "What's next in the queue?"
Activity-first
"I need to finish this design" / "Let me get back to writing"
2. Map What Triggers a Session
Categorize why users open the product. The trigger tells you what the user is optimizing for when they first arrive — which is exactly what primary navigation needs to serve.
Process-first
A recurring ritual — "every Monday I do X"
Entity-first
A status check — "I need to see where this thing stands"
Queue-first
A notification or alert — something happened and needs a response
Activity-first
A block of time — "I have two hours to create"
3. Look at Session Shape
Observe the pattern of a typical session. If you have analytics, session duration and page-transition patterns reveal this empirically. If not, five minutes of observation will usually do it.
Process-first
Linear or semi-linear — enter a phase, do the work, move to the next phase
Entity-first
Deep on one object — find the thing, examine it from multiple angles, act on it
Queue-first
Short bursts across many items — scan, pick, process, next, next, next
Activity-first
Long immersive stretches — enter focused mode, stay there, minimal navigation
4. The Blank Screen Test
Forget the current UI entirely. If the user opened a blank screen and could type or say what they want, what would they ask for? This strips away the bias of how the product is currently built and gets to underlying intent.
Process-first
"Start my month-end close"
Entity-first
"Take me to Project Alpha"
Queue-first
"Show me my tickets"
Activity-first
"Open the editor"
Breaking a Tie

If two models feel equally strong after running the diagnostics, apply these tiebreakers in order: 1. Default to whichever model reduces the most navigation steps for the user's single most frequent task. 2. Lean toward the model that new users grasp faster — a faster mental model means shorter onboarding. 3. If the tie persists, that's a signal: you may need a hybrid with co-equal primary anchors (e.g., a sidebar with both "Queue" and "Accounts" as peer items) rather than forcing one model to dominate.

Mapping Principles (Apply to All Models)

Persistence mirrors stability. Whatever occupies your primary navigation should be the most stable element of the experience. It should rarely change between releases.

Screen real estate mirrors scope. The primary anchor earns the most persistent screen space. Secondary concerns live inside that anchor's territory. In-page interactions live inside the page.

Interaction cost mirrors commitment. Switching between primary-level items is a significant context shift. Switching between secondary items should be lightweight. In-page activities should require near-zero navigation.

The Persistence Test

If a navigation element disappears when it shouldn't, or persists when it's irrelevant, you've got a level mismatch. Legitimate exceptions: immersive modes hide primary nav deliberately; queue-first detail views may collapse the queue to a panel.

Handling Cross-Job Tasks: Job Gravity

Tasks that serve multiple Jobs need a home in the navigation. Three approaches:

ApproachHow it worksBest when...Tradeoff
DuplicateTask appears under multiple Jobs as separate entries.The Jobs have genuinely different workflows around the same Task.Users distrust the nav. Maintenance doubles.
Single homeTask lives under one Job only.Usage from secondary Job is negligible (<5%).Secondary-Job users can't find it.
Job GravitySingle canonical home under the most-used Job. Pointers from other Jobs.Multiple Jobs legitimately need the Task.Pointer mechanisms add work.

Job Gravity is the recommended default. The four pointer mechanisms: 1. Cross-link in secondary nav. 2. Contextual shortcut inline. 3. Search / ⌘K result. 4. User-pinned favorite.

Step 03

Scope

What are my product's navigation constraints?

Before assessing behavioral complexity or selecting paradigms, profile your product across six dimensions. Your answers directly constrain which patterns will work and which will break.

1. How many top-level Jobs does your product support?
≤5 Jobs: Expose all in a single layer — top bar or bottom tabs.
5–15 Jobs: Side navigation with groupings, or a hub with categorized entry points. Consider ⌘K as an accelerator.
15+ Jobs: Compound approach — hub-and-spoke plus search, or role-based filtering.
2. How deep is your Task tree?
1 level: Tabs or segmented controls within each Job. No breadcrumbs needed.
2 levels: Collapsible sidebar sections. Consider breadcrumbs as a supplement.
3+ levels: Sidebar for top two levels; breadcrumbs for the rest. Without breadcrumbs, users will get lost.
3. How varied is your user expertise?
Uniform: One navigation system. Optimize for the dominant expertise level.
Moderate spread: Structural nav for everyone, ⌘K for power users. Role-aware defaults help.
Extreme spread: Layered navigation — visible structural nav for discovery, keyboard shortcuts for speed, guided flows for onboarding.
4. How concentrated is usage?
Highly concentrated (~80% of sessions involve 2–3 Tasks): Elevate those Tasks to Job-level prominence.
Evenly distributed: Navigation needs equal weight across many options — side nav or hub patterns.
Mixed (daily + quarterly Tasks): Quarterly Tasks can't be buried. Search and ⌘K become safety nets.
5. How much navigation learning will users tolerate?
Near-zero: Must be immediately obvious. Familiar patterns only — top tabs, bottom bars, visible labels.
Moderate: Users will invest time if the product is central to their work. Richer patterns with good onboarding.
High: Mission-critical, daily use. Training budgets exist. Optimize for power-user efficiency.
6. What platforms does your product serve?
Desktop-only: Full access to side nav, hover states, keyboard shortcuts, multi-panel layouts.
Responsive web: Your composition formula needs a responsive strategy. Side nav collapses; hover interactions need touch alternatives.
Native mobile: Platform conventions apply. iOS favors bottom tabs; Android favors top tabs and drawers. ⌘K becomes pull-down search.
Multi-platform: Two composition formulas sharing a JTBD hierarchy but using different paradigms per platform.
What to do with your answers

Your profile across these six dimensions acts as a filter for Step 05. Each answer narrows which paradigms are viable. Here's how they combine in practice:

Example: A B2B analytics platform.

How many top-level Jobs? → 8
How deep is your Task tree? → 2 levels
How varied is your user expertise? → Extreme spread
How concentrated is usage? → 3 Tasks dominate
How much learning tolerance? → Moderate
What platforms? → Responsive web

Reading across: side navigation handles the 8 Jobs (too many for top nav), collapsible sections handle the 2-level depth, ⌘K serves power users while visible labels serve new ones, the 3 high-frequency Tasks earn pinned shortcuts, and the responsive constraint means you'll need a collapse strategy for the sidebar on smaller screens.

The key is to read your answers together, not in isolation. A single dimension rarely rules out a pattern on its own — it's the combination that narrows the field.

Step 04

Assess

How do roles and context shape the navigation?

Three behavioral dimensions constrain your paradigm options: role-based flexing at the Job level, adaptive flexing at the Task and Activity levels, and user context that shapes how every tier should feel.

Role-Based Navigation

In Step 01, you defined Job maps for each persona and calculated their overlap ratio. That ratio directly determines which approach to take here — use it as your guide when reading the table below.

ApproachHow it worksWhen to useOverlapRisk
Separate appsEach role gets its own application.Roles share <20% of Jobs.<30% shared JobsMaintenance doubles.
Role-filteredOne nav, items shown/hidden by role.Roles share core Jobs but differ in access.30–70% shared JobsGhost nav — users sense hidden items.
Role-aware defaultsSame nav, different starting points by role.Same Jobs, different frequency patterns.>70% shared JobsLowest risk.
Recommendation

Start with role-aware defaults. Only escalate to role-filtered when Jobs genuinely diverge. Only build separate apps when Job maps have less than 20% overlap. When roles share more than 70% of Jobs, treat them as one role with different defaults — a PM lands on a project dashboard; a team lead lands on the team view. Same nav, different starting points.

Adaptive / Contextual Navigation

TypeLevelHow it worksExampleRisk
Workflow-stateActivityNav changes based on workflow progress.Editor toolbar shifts from editing to review tools."Where did my button go?"
Frequency-adaptiveTaskFrequently-used items gain prominence.Slack reorders channels by recent activity.Spatial memory breaks.
Data-state-drivenTaskSystem data changes navigation priority.Overdue SLA ticket rises to queue top. Blocked issue surfaces a warning.Loss of user control if overused.
Mode-switchingJobEntire nav restructures on mode change.Figma: file browser → canvas.Two mental models.
Data-State-Driven Priority

Distinct from frequency-adaptive. Frequency responds to what the user does often. Data-state responds to what the system says is urgent now — SLA breaches, blocked dependencies, overdue tasks. Use sparingly: if everything is urgent, nothing is.

The Cardinal Rule

Never remove navigation options silently. If something will disappear, move, or change, the user needs to understand why. Adaptive navigation must be transparently adaptive.

User Contexts

The context a user brings to a Task — whether they're uncertain, expert, exploring, or under pressure — changes how navigation should behave, even when the structural mapping stays the same.

Anxiety / First-time uncertainty

What it feels like: Fear of making a mistake, uncertainty about consequences.

Navigation implication: Narrow the navigation. Reduce choices. Guide with steppers and linear flows.

Example: Stripe's onboardingStripe collapses the full dashboard into a focused checklist — a guided stepper that isolates the user from complexity.

Confidence / High-frequency flow

What it feels like: Mastery, impatience with unnecessary steps. "I know exactly what I need."

Navigation implication: Minimize structural navigation. Provide keyboard shortcuts, ⌘K, and direct-action buttons that bypass hierarchy.

Example: Salesforce Global ActionsThe "+" button lets experienced reps create a record from anywhere, bypassing hierarchy entirely.

Exploration / Discovery

What it feels like: Uncertainty about what questions to ask. "I don't know what I don't know."

Navigation implication: Expose multiple entry points with enough context to help self-selection. Hub layouts, visible tab bars, and card-based overviews.

Example: Google Analytics 4 HomeGA4's home screen shows summary cards — each a window into a different dimension — helping users self-select.

Urgency / Crisis response

What it feels like: Pressure, tunnel vision. "Something is broken and I need to fix it now."

Navigation implication: Strip away non-essential options. Surface critical Tasks in one click from anywhere.

Example: PagerDuty's Incident ViewPagerDuty narrows the entire UI to the active incident — timeline, responders, runbooks, status. Normal navigation recedes.
Connecting Context to Structure

The JTBD hierarchy tells you what tier of navigation something belongs to. User context tells you how that tier should behave. When selecting paradigms in Step 05, test your choices against these four contexts: does this paradigm serve my anxious users? My confident users? My explorers? My urgent users?

Composing Your Navigation System

Individual paradigms are building blocks. A well-composed system combines them across four slots. Tension arises when two paradigms compete for the same slot.

The Composition Formula

[Primary] + [Secondary] + [Depth] + [Accelerators]

One paradigm per structural slot. Two in the same slot = conflict. Empty primary or secondary = gap. Depth and accelerators are optional.

Composition Examples

Salesforce (entity-first): Side nav + Tabs + Breadcrumbs + ⌘K

Instagram (process-first): Bottom tab bar + Full-screen pages + None + Contextual actions

Figma (activity-first): Mode-switch + Side panel + None + ⌘K + Contextual toolbar

Zendesk (queue-first): Work queue + Tabs + None + Macros + Keyboard shortcuts

Primary Structure

Process-first: ≤5 Jobs → top nav. 5–15 → side nav. 15+ → hub & spoke.
Entity-first: Side nav or list view.
Queue-first: Work queue with thin sidebar for view switching.
Activity-first: The mode-switch mechanism itself.

Secondary Structure

Tab bar: 3–5 parallel options within the primary anchor.
Collapsible sub-sections: 5+ items.
Mutually exclusive at the same level.

Depth Handling (conditional)

Only needed beyond 2 levels. Breadcrumb + drill-down is the standard depth handler. Skip this for shallow products.

Accelerators (additive)

⌘K for users who know what they want. Contextual actions for Activity-level interactions. Both can coexist. Neither counts toward structural complexity.

Step 06

Audit

How do I know my navigation works?

Seven principles for auditing your navigation system. Each includes conceptual framing, a decision question, and an implementation example.

Conceptual (Designers, PMs)
Decision (Practitioners)
Implementation (Developers)

Principle 1: Frequency Overrides Hierarchy

Conceptual

If 80% of sessions involve the same Task, that Task earns Job-level prominence — regardless of logical position.

Decision

Question: Are your 3 highest-frequency Tasks reachable in ≤1 click?

Heuristic: >40% of sessions = persistent shortcut.

Implementation

"Create Invoice" lives under Finance → Billing → Invoices, but it's the most-used Task. Give it a "+" button in the header, a ⌘K shortcut, and a pinned sidebar entry.

Principle 2: Stability Matches Persistence

Conceptual

Top nav should change as rarely as your Jobs — almost never. In-page controls can change every release.

Decision

Question: Has top-level nav changed in 6 months?

Heuristic: Primary nav ≤1x/year. Secondary quarterly. In-page every sprint.

Implementation

GitHub's top nav (Code, Issues, PRs, Actions, Projects) has been stable for years. New features add within sections, not alongside them.

Principle 3: Context Must Be Preserved

Conceptual

Switching Tasks within a Job should preserve state. It should feel like switching tabs, not closing one app and opening another.

Decision

Question: Task A → Task B → Task A — is state preserved?

Heuristic: If switching causes data loss or scroll reset, it's broken.

Implementation

Notion: switching database views preserves filters, sort order, and scroll position. The lens changes; the context persists.

Principle 4: Progressive Disclosure, Not Confusion

Conceptual

Don't expose Activity-level nav until the user commits to a Task. Each tier reveals the next. But the user must always see which Job and Task they're inside.

Decision

Question: At the deepest point, can the user tell which Job and Task they're inside?

Heuristic: If only a page title is visible — add breadcrumbs.

Implementation

Stripe: sidebar shows Job section, header shows Task entity, breadcrumbs show depth. All three tiers visible simultaneously.

Principle 5: Escape Velocity

Conceptual

From any point, a user should reach any higher level in ≤2 interactions.

Decision

Question: How many clicks from deepest point to Job level?

Heuristic: ≤2 = good. 3 = okay with breadcrumbs. 4+ = broken.

Implementation

Three escape mechanisms: persistent sidebar, un-truncated breadcrumbs, command palette. Most complex products need at least two.

Principle 6: Navigation as Wayfinding

Conceptual

At every moment, navigation must answer three questions. If any is unanswerable, the system is broken.

Decision

Three questions your navigation must answer at every moment:

1. Where am I? → Active states, breadcrumbs, page titles.

2. Where can I go? → Visible nav options, available actions.

3. How do I get back? → Back buttons, breadcrumbs, persistent nav.

If any answer is unclear at any point, the navigation system is broken.

Implementation

"Where am I?" — Active sidebar highlighted. "Where can I go?" — Sibling Tasks as tabs. "How do I get back?" — Clickable breadcrumb, persistent sidebar, working browser back.

Principle 7: Navigation Must Feel Instant

Conceptual

Structural correctness means nothing if navigation feels slow. Every transition should feel like it happens in response to the user's action, not after it.

Decision

Question: Do your three highest-frequency transitions complete in under 200ms perceived time?

Heuristic: Primary switches <100ms. Secondary <300ms. Depth transitions <500ms with a loading indicator within 100ms.

Implementation

Optimistic UI for tab switches. Preload adjacent views on hover. Skeleton screens for deeper transitions. Linear and Notion pre-cache sidebar items so page switches feel instant.

Final Heuristic

Map your Jobs and Tasks on paper first. Count the Jobs. Measure the Task depth. Note the frequency patterns. Identify your navigation anchor. Then pick paradigms and compose them. The hierarchy dictates the paradigm, never the reverse.

Using this framework: Follow the six steps in order. Define (1), Map (2), Scope (3), Assess (4), Select (5), Audit (6). Resist jumping to paradigm selection — the steps before it are what make the selection defensible.

Reference

Common Questions

Edge cases and ambiguities that come up when applying the framework to real products.

A Task looks the same in two different Jobs — do I need Job Gravity?

It depends on whether the user is asking the same question or a different question.

Same Task, multiple Jobs: "Generate a financial report" is the same work from either Stakeholder Reporting or Compliance. Job Gravity applies: pick a canonical home, add pointers.

Same Activity, different Task context: "Review analytics" during Discovery and during Measurement involve the same screen but different questions. These are different Tasks — design each view for its context. No Job Gravity needed.

How do I handle navigation for AI-assisted features?

Treat AI/automated items as a layer that sits alongside structural navigation, not within it. A "Suggested" section in a queue, an AI sidebar panel, or inline suggestions within a Task view are valid patterns.

The key principle: AI-surfaced items should never replace or reorder structural navigation without user consent. They're accelerators — additive, not structural.

Appendix

Origins & Further Reading

This framework draws from two intellectual traditions.

Jobs to Be Done (The Job Level)

JTBD theory holds that users "hire" products to make progress. The Job is the fundamental unit — stable, emotionally resonant, product-independent.

Clayton Christensen popularized JTBD for innovation strategy. Tony Ulwick (Outcome-Driven Innovation) systematized it into a quantitative methodology. Bob Moesta (The Switch Framework) focuses on emotional dimensions — his work directly informs the user context section in Step 04.

Hierarchical Task Analysis (The Task & Activity Levels)

HTA (Annett & Duncan, 1967) decomposes work into goals, sub-goals, operations, and plans. Activity Theory (Kaptelinin & Nardi) extends this with attention to tools and social dynamics.

Christensen — "Know Your Customers' Jobs to Be Done" (HBR, 2016)
Ulwick — ODI overview at strategyn.com · Jobs to Be Done: Theory to Practice
Moesta — demandside.com · Demand-Side Sales 101
HTA — Annett & Duncan (1967) · Stanton, Handbook of Human Factors and Ergonomtic Methods
Activity Theory — Kaptelinin & Nardi, Acting with Technology (MIT Press, 2006)

TL;DR

At a Glance

Navigation is the single most persistent element of any software product. When it works, people get things done. When it doesn't, they get lost, call support, and lose trust. Most navigation problems trace back to one root cause: the structure was designed around the product's internals instead of the user's intent.

Sightline is a six-step framework that fixes this by starting with what users are trying to accomplish and working outward to navigation structure. The core model defines user intent at three levels:

Jobs — the outcomes users care about (stable, role-tied). Tasks — the plannable units of work that advance a Job. Activities — the concrete UI interactions within a Task. Each level maps to a tier of navigation: primary, secondary, and in-page.

The six steps: Define user Jobs, Tasks, and Activities. Map them to navigation tiers. Scope your product's constraints. Assess role and context dimensions. Select from eight navigation paradigms. Audit against seven principles.

The outcome: navigation decisions that are defensible, user-centered, and durable — rather than arbitrary, org-chart-driven, or subject to redesign every quarter.
