Companion to the Sightline Framework — by Samiya Ali

1. JTBD Hierarchy

See Step 01: Define →

List every Job your users have, then break each into its constituent Tasks. Annotate each Task with how often it happens and how critical it is — these annotations drive navigation placement decisions in later steps.

Example
Job: Keep my team's financial health strong
 Task: Create monthly budget reports — Monthly / Blocking
 Task: Review expense approvals — Daily / Important

2. Navigation Anchor

See Step 02: Map →

Based on your four diagnostic checks (Language, Session Trigger, Session Shape, Blank Screen), identify which mental model dominates how your users navigate their work.

Example
Primary: Process-First (3 of 4 diagnostics). Session Trigger and Session Shape both pointed to sprint phases. Language was split between entity and process.

Secondary Anchor

Weighted Scores

3. Scope Profile

See Step 03: Scope →

Answer six constraint questions about your product. The combination of answers filters which navigation paradigms are viable and flags dimension conflicts to resolve in Step 05.

Example
4 Jobs · 2-level depth · Moderate expertise · Concentrated usage · Responsive web · Moderate tolerance

4. Role & Context Decisions

See Step 04: Assess →

Record how you're handling multiple user roles (if applicable) and which adaptive navigation patterns you're applying. Use the overlap ratio to determine whether roles share a navigation structure or need separation.

Example
Role-aware defaults. 70% shared Jobs across Admin and Member. Data-state adaptive for blocked items only (empty states trigger guided flows).

5. Paradigm Composition

See Step 05: Select →

Choose one paradigm for each structural slot: Primary, Secondary, Depth, and Accelerators. Record why each was chosen over alternatives — this rationale becomes the reference when stakeholders question the architecture.

Example
[Side nav] + [Tab bar] + [Breadcrumbs] + [&#8984;K]. Side nav chosen over top nav because responsive collapse is straightforward with 4 Jobs.

Mobile Composition Formula

6. Resulting Navigation Structure

See Step 05: Navigation Outline →

Write out your navigation structure as an indented outline. This is the deliverable that feeds Step 06's audit and becomes the reference artifact for your team.

Example
Primary (left rail): Standup (pinned) · Board · Backlog · Reports · Retrospectives
Secondary (tabs within Board): Board | List | Timeline
Depth: Active Sprint → Story-123 → Subtask
Accelerator: ⌘K for ticket lookup by ID

7. Known Risks & Open Questions

See Step 06: Audit →

Record close-call decisions, audit results that flagged issues, and any data dependencies you're missing. These become the focus areas for usability testing.

Example
Retrospectives may feel lost in primary rail given quarterly frequency — monitor whether users can find it. Standup pinning may confuse users who don't do daily standups.
